## Objective

A notification generated by M.P.A. must reliably:

1. Reach the intended user  
2. Be tappable  
3. Open the correct screen  

Proven on **real** Desktop / Android / iPhone (within Apple-supported web push capabilities).

> Do not assume anything works. Verify every layer.

## What this is / is not

| Is | Is not |
| --- | --- |
| End-to-end forensic certification | Adding new notification categories for product expansion |
| Fixing broken delivery / deep links / enrollment | Redesigning NotificationService architecture (API-001 stands) |
| Master Admin diagnostics + self-heal (scoped) | Native iOS/Android SDKs (web/PWA only unless separately approved) |
| Real-device evidence package | Certifying from staging-only or simulated sends |

## Architecture (current — API-001)

```
Domain event
  → notify()  [NotificationService]
    → preferences / quiet hours / recipients
    → insert in_app_notifications
    → NotificationProvider.send (OneSignal)
      → absolute launch URL
      → include_subscription_ids
  ← push_delivery_status on row
```

Client enroll path (API-001A):

```
Permission → OneSignal Web SDK → subscription id
  → POST /api/notifications/devices
  → resident_devices (+ prefs)
```

## Explicit non-goals

- Replacing OneSignal (ADR-017 stands)  
- Self-hosted VAPID web push  
- SMS as a push substitute
