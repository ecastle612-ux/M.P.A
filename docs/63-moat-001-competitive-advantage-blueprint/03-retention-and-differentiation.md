## 6. Customer retention strategy

### Why customers stay

| Reason | Mechanism |
| --- | --- |
| **Daily habit** | Morning opens on Today’s Work; muscle memory for ⌘K / Quick Add |
| **Interrupt recovery** | Resume due items without archaeology |
| **Team coordination** | Shared queues, assignments, bulk morning clears |
| **Resident/vendor already enrolled** | QR, portal, threads — leaving breaks the front door |
| **Automation lock-in** | Org-tuned rules + accepted AI patterns |
| **Knowledge accumulation** | Edited drafts, KB answers, vendor prefs |
| **Reporting continuity** | Owner narratives and history live here |
| **Migration already paid** | Switching cost was front-loaded once |

### Switching costs (ethical, product-real)

| Cost type | Detail |
| --- | --- |
| **Operational** | Retrain staff off OS grammar |
| **Data** | Export possible; *operational graph fidelity* hard to rehydrate |
| **Channel** | Residents/vendors must re-enroll elsewhere |
| **Automation** | Rebuild rules + AI feedback weights |
| **Reputation** | Mid-migration dual systems risk owner trust |

**Anti-pattern:** Dark patterns / hostage data. Retention must come from **superior daily ops**, with honest export paths (trust moat).

### Retention plays (post-Approve product)

1. Time-to-value &lt; 1 week for Design Partners (RC-001 constrained GO).  
2. Make Today’s Work the unmissable home.  
3. Measure “actions from OS surfaces” weekly.  
4. Grow org KB + AI accepts — visible “gets smarter” moments.  
5. Never force dual CRUD + guided paths (moat regression).

## 9. Automation differentiation

| Principle | Detail |
| --- | --- |
| **Automate the boring** | Categorize, remind, rank, draft |
| **Confirm the consequential** | Assign vendor, send message, money, leasing decisions |
| **Workflow-native** | Automations advance guided jobs; don’t spawn shadow processes |
| **Observable** | Every automation writes activity; dismissible suggestions |
| **Org policy** | Low-risk auto opt-in; high-risk never silent |

Vendor automation = **recommend + one-click confirm**, not silent dispatch, until trust metrics justify deeper policy engines.

## 10. Data network effects

M.P.A. does **not** pursue cross-tenant “global AI” that pools competitor customer PII.

| Effect | Scope | Why it retains |
| --- | --- | --- |
| **Operational graph density** | Single org | More linked entities → better Context Actions |
| **AI feedback weights** | Single org | Accept/dismiss improves ranking for *that* team |
| **Vendor performance history** | Single org | Better match suggestions over time |
| **Comms + KB corpus** | Single org | Faster drafts in the org’s voice |
| **Portfolio pattern memory** | Single org | Seasonal maintenance / delinquency priors |
| **Multiplayer habits** | Team | Shared Today’s Work norms |

**Marketplace-scale network effects** (cross-org vendor liquidity) are a **Year 3–5** option only with explicit privacy design — not Year-1 identity.
